## Q1: 请介绍常用数据结构及其时间复杂度

**规范答案：**

常用数据结构可分为线性结构和非线性结构：

**线性结构：**
- **数组**：连续内存，支持 O(1) 随机访问，插入/删除 O(n)
- **链表**：离散内存，插入/删除 O(1)（已知位置），查找 O(n)
- **栈**：后进先出（LIFO），push/pop 均 O(1)
- **队列**：先进先出（FIFO），enqueue/dequeue 均 O(1)

**非线性结构：**
- **哈希表**：基于哈希函数，平均 O(1) 查找/插入/删除，最差 O(n)
- **二叉搜索树**：平均 O(log n) 查找/插入/删除，最差退化为 O(n)
- **红黑树/AVL树**：自平衡 BST，保证 O(log n)
- **堆**：完全二叉树，插入/删除 O(log n)，取最值 O(1)
- **图**：由顶点和边组成，BFS/DFS 遍历 O(V+E)

**得分点：**
- [ ] 正确区分线性结构与非线性结构
- [ ] 每种数据结构的时间复杂度准确
- [ ] 理解数组与链表的本质区别（连续内存 vs 离散内存）
- [ ] 知道哈希表的冲突处理机制（链地址法 / 开放寻址）
- [ ] 了解树的平衡性对性能的影响

**加分项：**
- 提到跳表（Skip List）等高级数据结构
- 能结合实际场景说明选型（如 LRU 缓存用哈希表+双向链表）
- 提到 Java HashMap 的红黑树退化机制（链表长度 > 8）

**追问方向：**
- 哈希冲突的解决方案有哪些？各自优缺点？
- 红黑树和 AVL 树有什么区别？为什么 Java 选红黑树？
- 堆和优先队列的关系是什么？

## Q2: 常见排序算法的比较

**规范答案：**

| 算法 | 平均时间 | 最差时间 | 空间 | 稳定性 |
|------|---------|---------|------|--------|
| 冒泡排序 | O(n²) | O(n²) | O(1) | 稳定 |
| 选择排序 | O(n²) | O(n²) | O(1) | 不稳定 |
| 插入排序 | O(n²) | O(n²) | O(1) | 稳定 |
| 归并排序 | O(n log n) | O(n log n) | O(n) | 稳定 |
| 快速排序 | O(n log n) | O(n²) | O(log n) | 不稳定 |
| 堆排序 | O(n log n) | O(n log n) | O(1) | 不稳定 |

快排最常用，虽然最差 O(n²)，但通过随机化 pivot 可以避免，且缓存友好性好。
归并排序适合外排序和链表排序。
对于小规模或近似有序数据，插入排序效率最高。

**得分点：**
- [ ] 至少列出 4 种排序算法的复杂度
- [ ] 正确区分稳定与不稳定排序
- [ ] 解释快排最差情况及优化方法
- [ ] 知道不同场景下的排序选择

**加分项：**
- 提到 TimSort（Python/Java 默认排序）融合了归并和插入排序
- 能分析快排为何实际性能优于堆排序（缓存局部性）
- 提到计数排序/桶排序/基数排序等线性时间排序

## Q3: 请解释设计模式中的单例模式

**规范答案：**

单例模式确保一个类只有一个实例，并提供一个全局访问点。

**实现方式（以 Java 为例）：**

1. **饿汉式**：类加载时即创建，线程安全但可能浪费资源
2. **懒汉式**：首次使用时创建，需要加锁保证线程安全
3. **双重检查锁（DCL）**：结合懒加载和线程安全，需要 volatile 防止指令重排
4. **静态内部类**：利用类加载机制保证线程安全和懒加载
5. **枚举**：最简洁安全，防止反射和序列化破坏

**适用场景：**
- 数据库连接池
- 配置管理器
- 日志工具类
- 线程池

**得分点：**
- [ ] 清楚单例的核心目的（唯一实例 + 全局访问）
- [ ] 至少说出 3 种实现方式
- [ ] 知道 DCL 为什么需要 volatile
- [ ] 了解线程安全问题

**加分项：**
- 提到枚举实现是《Effective Java》推荐方式
- 能说明反射和序列化如何破坏单例及防御方式
- 分析 Spring 中 Bean 的单例实现（IoC 容器管理）

**追问方向：**

## Q4: TCP 三次握手和四次挥手的过程

**规范答案：**

**三次握手（建立连接）：**
1. 客户端 → 服务端：SYN=1, seq=x（客户端进入 SYN_SENT）
2. 服务端 → 客户端：SYN=1, ACK=1, seq=y, ack=x+1（服务端进入 SYN_RCVD）
3. 客户端 → 服务端：ACK=1, seq=x+1, ack=y+1（双方进入 ESTABLISHED）

**为什么是三次：** 防止已失效的连接请求报文突然到达服务端，导致错误连接。本质上是确认双方的发送和接收能力。

**四次挥手（断开连接）：**
1. 客户端 → 服务端：FIN=1, seq=u（客户端进入 FIN_WAIT_1）
2. 服务端 → 客户端：ACK=1, ack=u+1（服务端进入 CLOSE_WAIT，客户端进入 FIN_WAIT_2）
3. 服务端 → 客户端：FIN=1, seq=w（服务端进入 LAST_ACK）
4. 客户端 → 服务端：ACK=1, ack=w+1（客户端进入 TIME_WAIT，等待 2MSL 后关闭）

**为什么是四次：** 因为 TCP 是全双工，每个方向需要单独关闭，服务端收到 FIN 后可能还有数据未发完。

**得分点：**
- [ ] 正确描述三次握手每一步的标志位和状态变化
- [ ] 解释为什么不能两次握手
- [ ] 正确描述四次挥手过程
- [ ] 知道 TIME_WAIT 状态存在的原因和 2MSL 等待

**加分项：**
- 提到 SYN Flood 攻击和 SYN Cookie 防御
- 解释半连接队列和全连接队列
- 提到 TCP Fast Open（TFO）优化

**追问方向：**
- TIME_WAIT 过多会有什么问题？如何解决？
- TCP 和 UDP 的区别？各适用什么场景？
- TCP 的拥塞控制机制是怎样的？

## Q5: 进程、线程和协程的区别

**规范答案：**

| 特性 | 进程 | 线程 | 协程 |
|------|------|------|------|
| 定义 | 操作系统资源分配的基本单位 | CPU 调度的基本单位 | 用户态轻量级线程 |
| 内存 | 独立地址空间 | 共享进程地址空间 | 共享线程栈 |
| 切换开销 | 大（涉及内存映射等） | 中（内核态切换） | 小（用户态切换） |
| 通信方式 | 管道/消息队列/共享内存/信号量 | 共享变量（需同步） | 通过 channel 或 yield |
| 创建开销 | MB 级别 | KB~MB 级别 | KB 级别 |
| 并行/并发 | 可真并行 | 可真并行 | 并发（单线程内） |

**得分点：**
- [ ] 正确区分进程和线程的核心差异（资源分配 vs CPU调度）
- [ ] 知道线程共享进程资源但有独立栈和寄存器
- [ ] 理解协程的用户态调度特性
- [ ] 了解 IPC 通信方式

**加分项：**
- 结合语言举例（Go goroutine、Python asyncio、Java 虚拟线程）
- 分析多进程 vs 多线程 vs 协程的适用场景
- 提到 Go 的 GMP 调度模型

**追问方向：**

## Q6: HTTP 和 HTTPS 的区别？HTTPS 的握手过程

**规范答案：**

**核心区别：**
- HTTP 明文传输，HTTPS = HTTP + TLS/SSL 加密传输
- HTTP 默认端口 80，HTTPS 默认端口 443
- HTTPS 需要 CA 证书

**HTTPS（TLS 1.2）握手过程：**
1. Client Hello：客户端发送支持的 TLS 版本、加密套件列表、随机数 Client Random
2. Server Hello：服务端选择加密套件，返回随机数 Server Random + 数字证书
3. 客户端验证证书（CA 链验证），生成 Pre-Master Secret，用服务端公钥加密后发送
4. 双方根据 Client Random + Server Random + Pre-Master Secret 生成会话密钥
5. 后续通信使用对称加密（会话密钥）

**得分点：**
- [ ] 明确 HTTPS = HTTP + TLS
- [ ] 知道非对称加密用于密钥交换，对称加密用于数据传输
- [ ] 描述证书验证的基本流程
- [ ] 了解随机数的作用

**加分项：**
- 提到 TLS 1.3 简化为 1-RTT 甚至 0-RTT
- 能解释中间人攻击和证书固定（Certificate Pinning）
- 理解前向安全性（Forward Secrecy）和 ECDHE 密钥交换

**追问方向：**
- 对称加密和非对称加密各有什么优缺点？
- 什么是中间人攻击？如何防御？
- HTTP/2 和 HTTP/3 有什么改进？
