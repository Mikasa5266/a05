## Q1: 微服务架构与单体架构的区别？什么时候选择微服务？

**规范答案：**

| 维度 | 单体架构 | 微服务架构 |
|------|---------|-----------|
| 部署 | 整体打包部署 | 独立部署、独立扩展 |
| 技术栈 | 统一技术栈 | 各服务可用不同语言/框架 |
| 开发效率 | 初期快，后期慢 | 初期慢（基建成本），后期快 |
| 故障影响 | 一处故障可能全局影响 | 故障隔离，影响局部 |
| 运维复杂度 | 低 | 高（服务治理、链路追踪、分布式事务） |
| 数据管理 | 共享数据库 | 每个服务独立数据库 |

**适合微服务的场景：**
- 团队规模大（>10 人），需要独立开发和部署
- 业务复杂度高，有明确的领域边界
- 需要不同模块独立扩展（如订单模块流量远大于用户模块）
- 技术栈需要多样化

**不适合微服务的场景：**
- 初创项目、团队小、业务简单
- 没有足够的 DevOps 基建能力

**得分点：**
- [ ] 正确对比单体和微服务的核心差异
- [ ] 能说出微服务的优缺点
- [ ] 给出合理的选型依据（不是盲目追求微服务）
- [ ] 了解微服务带来的挑战（分布式事务、数据一致性）

**加分项：**
- 提到 DDD（领域驱动设计）指导服务拆分
- 提到服务网格（Service Mesh / Istio）
- 提到 Strangler Fig 模式进行渐进式拆分

**追问方向：**
- 微服务之间如何通信？RPC vs REST vs 消息队列？
- 分布式事务如何保证？了解 Saga 模式吗？
- 服务发现和注册是什么？有哪些实现方案？

## Q2: 消息队列的作用和使用场景？Kafka 和 RabbitMQ 的区别？

**规范答案：**

**消息队列的三大核心作用：**
1. **异步处理**：用户注册后异步发送邮件/短信，减少响应时间
2. **流量削峰**：秒杀场景中用 MQ 缓冲请求，保护下游服务
3. **服务解耦**：上游服务只需发消息，不关心下游谁消费

**Kafka vs RabbitMQ：**

| 特性 | Kafka | RabbitMQ |
|------|-------|----------|
| 定位 | 分布式流处理平台 | 传统消息代理 |
| 吞吐量 | 百万级 TPS | 万级 TPS |
| 消息模型 | 拉模式（Consumer 主动拉取） | 推模式（Broker 推送） |
| 消息存储 | 磁盘顺序写，可持久化 | 内存为主 |
| 消费模式 | Consumer Group，分区消费 | 多种交换机模式（direct/fanout/topic） |
| 适用场景 | 日志收集、大数据流、事件溯源 | 业务消息、任务队列、复杂路由 |

**得分点：**
- [ ] 说出消息队列的三大作用
- [ ] 用具体业务场景举例
- [ ] 知道 Kafka 和 RabbitMQ 的核心差异
- [ ] 了解消息丢失和重复消费问题

**加分项：**
- 提到消息可靠性保障（生产确认、持久化、消费确认）
- 分析如何保证消息顺序性
- 提到 RocketMQ 的延迟消息和事务消息

**追问方向：**
- 如何保证消息不丢失？
- 如何处理消息重复消费（幂等性）？
- Kafka 如何实现高吞吐？分区机制是怎样的？

## Q3: 如何设计一个高并发的秒杀系统？

**规范答案：**

**核心思路：** 分层削峰、缓存前置、异步处理、兜底降级

**前端层：**
- 按钮置灰防止重复点击
- 静态页面 CDN 缓存
- 请求加密 + 验证码防刷

**网关层：**
- 限流（令牌桶 / 滑动窗口）
- 请求去重
- 用户级别限频

**服务层：**
- Redis 预扣减库存（使用 Lua 脚本保证原子性）
- 库存扣减成功后发送 MQ 消息
- 异步处理订单创建和支付

**数据层：**
- 库存预热到 Redis
- 数据库乐观锁更新（version 字段）
- 分库分表分散热点

**兜底方案：**
- 服务熔断降级（Sentinel / Hystrix）
- 队列满时直接返回"已售罄"
- 监控告警

**得分点：**
- [ ] 有分层设计思维（前端 → 网关 → 服务 → 数据）
- [ ] 提到限流和削峰方案
- [ ] 使用 Redis 预扣减库存
- [ ] 异步下单避免数据库压力
- [ ] 有兜底和降级方案

**加分项：**

## Q4: RESTful API 设计原则

**规范答案：**

**核心原则：**
1. **资源导向**：URL 表示资源，使用名词而非动词（`/users` 而非 `/getUsers`）
2. **HTTP 方法语义化**：GET 查询、POST 创建、PUT 全量更新、PATCH 部分更新、DELETE 删除
3. **状态码规范**：200 成功、201 创建成功、400 参数错误、401 未认证、403 无权限、404 资源不存在、500 服务端错误
4. **版本控制**：URL 路径 `/api/v1/` 或 Header `Accept: application/vnd.api+json;version=1`
5. **分页与过滤**：`?page=1&size=20&sort=created_at:desc`

**响应格式统一：**
```json
{
  "code": 200,
  "message": "success",
  "data": { ... },
  "timestamp": 1709712000
}
```

**得分点：**
- [ ] 知道 REST 是资源导向的
- [ ] HTTP 方法与 CRUD 操作的对应关系
- [ ] 了解常用状态码
- [ ] 提到版本控制

**加分项：**
- 提到 HATEOAS（超媒体驱动）
- 对比 RESTful 和 GraphQL
- 提到幂等性设计

**追问方向：**
- PUT 和 PATCH 的区别？
- 如何设计幂等接口？
- GraphQL 相比 REST 有什么优缺点？

## Q5: Docker 和 Kubernetes 的基本概念

**规范答案：**

**Docker：**
- 容器化技术，将应用及其依赖打包为镜像（Image），运行为容器（Container）
- 核心概念：Dockerfile → 镜像 → 容器 → 仓库（Registry）
- 优势：环境一致性、轻量（共享宿主机内核，比 VM 轻量）、快速启动
- 核心命令：`docker build`、`docker run`、`docker compose`

**Kubernetes（K8s）：**
- 容器编排平台，负责容器的部署、扩缩容、服务发现和负载均衡
- 核心概念：
  - **Pod**：最小部署单元，包含一个或多个容器
  - **Service**：暴露 Pod 的稳定网络访问入口
  - **Deployment**：管理 Pod 的副本数和滚动更新
  - **ConfigMap/Secret**：配置和密钥管理
  - **Ingress**：外部流量入口和路由规则

**得分点：**
- [ ] 理解 Docker 容器 vs 虚拟机的区别
- [ ] 知道镜像分层和 Dockerfile 基本写法
- [ ] 了解 K8s 的 Pod、Service、Deployment 概念
- [ ] 理解容器编排解决的核心问题

**加分项：**
- 有实际使用经验（CI/CD 流水线中使用 Docker）
- 了解 K8s 的健康检查（liveness/readiness probe）
- 提到 Helm Chart 和 GitOps

**追问方向：**

## Q6: 认证授权方案：Session、JWT、OAuth 2.0

**规范答案：**

**Session 认证：**
- 服务端存储会话信息，客户端通过 Cookie 携带 Session ID
- 优点：服务端可主动失效，安全性较高
- 缺点：分布式环境需要 Session 共享（Redis 存储）

**JWT（JSON Web Token）：**
- 自包含的 Token（Header.Payload.Signature），服务端无需存储
- 优点：无状态、适合分布式和移动端
- 缺点：无法主动失效（需要黑名单机制）、Token 较大
- 结构：Header（算法类型）+ Payload（数据）+ Signature（签名验证）

**OAuth 2.0：**
- 第三方授权协议，核心角色：资源拥有者、客户端、授权服务器、资源服务器
- 主要授权模式：授权码模式（最安全）、隐式模式、密码模式、客户端凭证模式
- 典型场景：第三方登录（微信、GitHub）

**得分点：**
- [ ] 清楚 Session 和 JWT 的核心区别（有状态 vs 无状态）
- [ ] 了解 JWT 的三段结构和签名原理
- [ ] 知道 OAuth 2.0 的基本流程
- [ ] 理解各方案的适用场景

**加分项：**

- 分析 JWT 的安全风险（密钥泄露、Token 窃取）和防御方案
- 提到 RBAC / ABAC 权限模型

**追问方向：**
