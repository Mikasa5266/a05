## Q1: MySQL 索引的底层数据结构是什么？为什么用 B+ 树？

**规范答案：**

MySQL InnoDB 引擎使用 **B+ 树**作为索引的底层数据结构。

**B+ 树的特点：**
- 非叶子节点只存键值不存数据，单个节点能容纳更多键值，树更矮
- 所有数据都存储在叶子节点，且叶子节点用双向链表连接
- 查询复杂度稳定为 O(log n)

**为什么不用其他结构：**
- **哈希表**：不支持范围查询和排序（等值查询可以用 Memory 引擎的 Hash 索引）
- **B 树**：非叶子节点也存数据，导致树更高、磁盘 IO 更多
- **红黑树**：树高度大，不适合磁盘存储（每次访问一个节点就是一次磁盘 IO）
- **跳表**：范围查询性能不如 B+ 树，且空间占用较大

**聚簇索引 vs 二级索引：**
- 聚簇索引：叶子节点存储完整行数据，InnoDB 主键索引即聚簇索引
- 二级索引：叶子节点存储主键值，查询需要回表（先查二级索引拿到主键，再查聚簇索引拿到行数据）

## Q2: MySQL 事务的四大特性（ACID）和隔离级别

**规范答案：**

**ACID 四大特性：**
- **原子性（Atomicity）**：事务中的操作要么全部成功，要么全部回滚。通过 undo log 实现。
- **一致性（Consistency）**：事务前后数据库保持一致性状态。是最终目的，由其他三个特性保证。
- **隔离性（Isolation）**：并发事务之间互不干扰。通过锁机制和 MVCC 实现。
- **持久性（Durability）**：事务提交后数据永久保存。通过 redo log 实现。

**四种隔离级别（从低到高）：**

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| READ UNCOMMITTED | ✓ | ✓ | ✓ |
| READ COMMITTED | ✗ | ✓ | ✓ |
| REPEATABLE READ（InnoDB 默认） | ✗ | ✗ | ✗* |
| SERIALIZABLE | ✗ | ✗ | ✗ |

*InnoDB 在 RR 级别通过 MVCC + 间隙锁（Gap Lock）基本避免幻读。

**MVCC 机制：**
通过隐藏字段（事务 ID、回滚指针）和 Read View 实现，读操作不加锁，提高并发性能。

## Q3: Redis 常用数据结构及应用场景

**规范答案：**

**五种基础数据结构：**

| 类型 | 底层实现 | 典型应用场景 |
|------|---------|-------------|
| String | SDS（简单动态字符串） | 缓存、计数器、分布式锁 |
| Hash | ziplist / hashtable | 对象属性存储（用户信息） |
| List | quicklist（ziplist + 链表） | 消息队列、最新列表 |
| Set | intset / hashtable | 标签、共同好友、抽奖 |
| Sorted Set | ziplist / skiplist + hashtable | 排行榜、延迟队列 |

**三种高级结构：**
- **HyperLogLog**：基数统计（如 UV 统计），仅 12KB 内存
- **Bitmap**：位操作，适合签到、在线状态
- **GEO**：地理位置，附近的人

## Q4: 缓存穿透、缓存击穿和缓存雪崩

**规范答案：**

| 问题 | 定义 | 解决方案 |
|------|------|---------|
| **缓存穿透** | 查询数据库和缓存都不存在的数据，请求直接打到数据库 | ① 缓存空值（设短 TTL）② 布隆过滤器拦截 ③ 参数校验 |

| **缓存雪崩** | 大量 key 同时过期或 Redis 宕机 | ① 过期时间加随机值 ② 多级缓存 ③ Redis 集群高可用 |

## Q5: SQL 优化的常见手段

**规范答案：**

**1. 索引优化：**
- 对 WHERE、JOIN、ORDER BY 字段建立合理索引
- 使用联合索引并遵循最左前缀原则
- 避免索引失效（函数操作、隐式类型转换、LIKE '%xx'、OR 条件）

**2. 查询优化：**
- 避免 SELECT *，只查需要的字段
- 用 EXPLAIN 分析执行计划，关注 type、key、rows、Extra 字段
- 深分页优化：用游标分页或延迟关联
- 小表驱动大表（IN vs EXISTS 选择）

**3. 表设计优化：**
- 选择合适的数据类型（够用即可，如用 INT 不用 BIGINT）
- 适当冗余减少 JOIN
- 大字段拆分（垂直分表）

**4. 架构优化：**
- 读写分离
- 分库分表（水平/垂直拆分）
- 引入缓存层

## Q6: 分布式锁的实现方案对比

**规范答案：**

**三种主流方案：**

| 方案 | 实现方式 | 优点 | 缺点 |
|------|---------|------|------|
| Redis | SET key val NX EX | 性能高、实现简单 | 主从切换可能丢锁 |
| ZooKeeper | 临时有序节点 | 强一致性、锁释放可靠 | 性能较低 |
| MySQL | FOR UPDATE / 唯一索引 | 简单可靠 | 性能最差 |

**Redis 分布式锁注意事项：**
1. 设置合理的过期时间，防止死锁
2. 使用唯一值标识锁持有者，释放时先比较再删除（Lua 脚本保证原子性）
3. 对于主从架构，可使用 Redlock 算法提高可靠性
4. 看门狗机制（Redisson）自动续期
