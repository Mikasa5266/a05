# Java 后端 - Redis 与缓存面试知识库

## 1. Redis 数据结构

### 5 种基础数据结构
| 类型 | 底层实现 | 典型应用 |
|------|----------|----------|
| String | SDS（简单动态字符串） | 缓存、计数器、分布式锁 |
| Hash | ziplist / hashtable | 对象属性存储 |
| List | quicklist（ziplist + 链表） | 消息队列、时间线 |
| Set | intset / hashtable | 标签、共同好友 |
| ZSet | ziplist / skiplist + hashtable | 排行榜、延迟队列 |

### 3 种高级数据结构（Redis 5.0+）
- **HyperLogLog**：基数统计（UV 统计），12KB 误差 0.81%
- **Bitmap**：位运算，签到打卡
- **Stream**：消息队列，支持消费组

### 标准答案要点（5星回答）
1. 5 种基础类型的底层实现和使用场景
2. ZSet 跳表的结构和查询效率 O(logN)
3. SDS 相比 C 字符串的优势（二进制安全、O(1) 获取长度、预分配）
4. 了解 Redis 6.0 多线程 IO 模型
5. 实际项目中的数据结构选型经验

## 2. 缓存三大问题

### 缓存穿透
**定义**：查询不存在的数据，请求穿过缓存直达数据库
**解决方案**：
1. 布隆过滤器（Bloom Filter）：在缓存前增加一层过滤，不存在的 key 直接拦截
2. 缓存空值：`SET key "" EX 60`，短期缓存空结果
3. 请求校验：接口层参数合法性校验

### 缓存击穿
**定义**：热点 key 过期瞬间，大量并发请求直击数据库
**解决方案**：
1. 互斥锁（SETNX）：只允许一个线程重建缓存
2. 逻辑过期：缓存不设 TTL，在 value 中记录逻辑过期时间，异步更新
3. 热点 key 永不过期

### 缓存雪崩
**定义**：大量 key 同时过期或 Redis 宕机，请求全部打到数据库
**解决方案**：
1. 过期时间加随机偏移：`TTL + random(0, 300)`
2. Redis 集群 + 哨兵保证高可用
3. 限流降级：Sentinel / Hystrix 熔断保护

### 标准答案要点（5星回答）
1. 区分三大问题的触发条件
2. 每个问题至少给出 2-3 种解决方案
3. 布隆过滤器的原理（多个哈希函数 + 位数组）
4. 有实际处理缓存问题的经验
5. 了解缓存与数据库的一致性方案

## 3. Redis 持久化

### RDB（快照）
- 全量快照，fork 子进程执行，利用 COW（Copy-On-Write）
- 触发：`SAVE`（阻塞）/ `BGSAVE`（后台）/ 配置自动触发
- 优点：文件紧凑、恢复快
- 缺点：可能丢失最后一次快照后的数据

### AOF（追加日志）
- 记录每条写命令，顺序追加到文件
- 三种同步策略：always（每条刷盘）/ everysec（每秒）/ no（OS 决定）
- AOF 重写：`BGREWRITEAOF`，用最少命令表达当前数据状态
- 优点：数据更安全
- 缺点：文件更大、恢复更慢

### Redis 4.0 混合持久化
- AOF 文件前半部分是 RDB 格式，后半部分是增量 AOF
- 兼具 RDB 的快速恢复和 AOF 的数据安全

### 标准答案要点（5星回答）
1. RDB 和 AOF 的原理和优缺点对比
2. 理解 fork + COW 机制
3. AOF 重写的触发和实现
4. 混合持久化的工作方式
5. 生产环境的持久化策略选型

## 4. Redis 集群方案

### 主从复制
- 全量同步：`PSYNC ? -1` → RDB + 缓冲区增量
- 增量同步：基于 repl_offset 和 repl_backlog

### 哨兵（Sentinel）
- 监控：每秒 PING 检测实例状态
- 自动故障转移：主节点不可用时选举新主
- 客户端发现：通过 Sentinel 获取当前主节点地址

### Redis Cluster
- 16384 个 hash slot 分布在多个节点
- 客户端路由：MOVED 重定向 / ASK 重定向
- Gossip 协议传播节点状态
- 不支持：多 key 操作跨 slot（需 hash tag 解决）

### 标准答案要点（5星回答）
1. 主从复制的全量 vs 增量同步流程
2. 哨兵选举算法（Raft 类似）
3. Cluster 的 slot 分配和客户端路由
4. 脑裂问题及 min-slaves-to-write 解决方案
5. 生产集群规模和运维经验

## 5. 分布式锁

### Redis 分布式锁实现
```
# 加锁
SET lock_key unique_id NX PX 30000

# 释放锁（Lua 脚本保证原子性）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

### Redlock 算法
1. 获取当前时间 T1
2. 依次尝试在 N 个（≥5）Redis 实例上获取锁
3. 统计在多数节点（≥N/2+1）成功获取的数量
4. 有效期 = 初始有效期 - (当前时间 - T1)
5. 多数节点获取成功且有效期 > 0 → 获取锁成功

### 标准答案要点（5星回答）
1. SET NX PX 的完整加锁方式
2. 释放锁必须用 Lua 脚本保证原子性
3. 了解锁续期（看门狗机制，Redisson 实现）
4. Redlock 的争议（Martin Kleppmann 的质疑）
5. 与 ZooKeeper 分布式锁的对比
