# Java 后端 - 项目经验深挖面试知识库

## 1. 项目介绍框架（STAR 法则）

### 标准回答结构
- **Situation（背景）**：项目是做什么的，服务多少用户，技术栈是什么
- **Task（任务）**：你负责的核心模块是什么
- **Action（行动）**：你做了什么技术决策，怎么实现的
- **Result（结果）**：带来了什么可量化的收益（性能提升 X%、QPS 从 A 提升到 B）

### 5星回答示例
"我负责的是一个电商平台的订单系统，日均处理 50 万笔订单。为了解决大促期间的性能瓶颈，我主导将订单库从单库拆分为 16 库 64 表，分片键选择 user_id，使用 ShardingSphere。拆分后核心接口 RT 从 200ms 降到 30ms，支撑了双十一峰值 3000 QPS 无故障。"

## 2. 常见深挖问题方向

### 系统设计类
- **如何设计一个秒杀系统？**
  - 标准答案要点：流量削峰（消息队列）、库存预扣减（Redis 原子操作）、限流（令牌桶/滑动窗口）、异步下单、页面静态化 + CDN
  
- **如何设计一个短链系统？**
  - 标准答案要点：Base62 编码生成短码、布隆过滤器去重、302 重定向、统计点击分析、缓存热门短链

- **如何设计一个消息推送系统？**
  - 标准答案要点：WebSocket 长连接、消息持久化、ACK 确认机制、离线消息存储、推拉结合

### 性能优化类
- 慢 SQL 如何排查？→ 慢日志 + EXPLAIN + 索引优化
- 接口超时怎么排查？→ 链路追踪（Skywalking/Zipkin）+ 火焰图
- 内存泄漏怎么定位？→ jmap dump + MAT 分析 + JVisualVM

### 高可用类
- 服务挂了怎么办？→ 限流熔断降级（Sentinel）+ 灰度发布 + 快速回滚
- 数据丢了怎么办？→ binlog 恢复 + 主从切换 + 异地多活

## 3. 微服务架构

### Spring Cloud 核心组件
| 组件 | 功能 | 对应方案 |
|------|------|----------|
| Nacos | 注册中心 + 配置中心 | 替代 Eureka + Config |
| OpenFeign | 声明式 HTTP 调用 | 替代 RestTemplate |
| Sentinel | 流量控制 + 熔断降级 | 替代 Hystrix |
| Gateway | API 网关 | 替代 Zuul |
| Seata | 分布式事务 | AT/TCC/Saga 模式 |
| SkyWalking | 链路追踪 | 替代 Zipkin |

### 微服务拆分原则
1. 单一职责：每个服务只做一件事
2. 高内聚低耦合：减少服务间依赖
3. 数据独立：每个服务有自己的数据库
4. 接口稳定：API 版本管理
5. 适当合并：避免过度拆分导致运维成本飙升

### 标准答案要点（5星回答）
1. 清楚说出使用的技术栈和选型原因
2. 微服务拆分的依据和边界划分
3. 服务间调用的容错处理（超时、重试、熔断）
4. 分布式事务的处理方案
5. 有实际的线上问题排查经验

## 4. 消息队列

### Kafka vs RocketMQ vs RabbitMQ
| 维度 | Kafka | RocketMQ | RabbitMQ |
|------|-------|----------|----------|
| 吞吐量 | 百万级 | 十万级 | 万级 |
| 消息可靠性 | 高（ISR + acks=all） | 高（同步刷盘+同步复制） | 高（镜像队列） |
| 延迟 | ms 级 | ms 级 | μs 级 |
| 顺序消息 | 分区有序 | 队列有序 | 队列有序 |
| 事务消息 | 支持 | 支持 | 不直接支持 |
| 适用场景 | 大数据日志 | 电商交易 | 中小型系统 |

### 消息可靠性保证（三阶段）
1. **生产端**：同步发送 + 失败重试 + 本地消息表
2. **Broker 端**：同步刷盘 + 多副本同步
3. **消费端**：手动 ACK + 幂等消费（唯一ID/状态机）

### 标准答案要点（5星回答）
1. 至少对比 2 种 MQ 的优缺点
2. 消息丢失的三个环节及解决方案
3. 消费幂等的实现方式
4. 消息积压的处理（扩容消费者、跳过非关键消息）
5. 有 MQ 在实际业务中的使用经验（如订单异步处理、数据同步等）
